RAG 챗봇 백엔드 상세 설계서

Retrieval-Augmented Generation (RAG) 기반 AI 챗봇의 백엔드 시스템 아키텍처

1. 서비스 개요

RAG 챗봇 서비스는 대규모 언어 모델(LLM)과 정보 검색 기술을 결합하여 사용자 질문에 대한 정확하고 맥락 기반의 답변을 제공하는 지능형 대화 시스템입니다.

1.1 서비스 특징

  • 정보 검색 기반: 내부 지식 베이스에서 관련 정보를 검색하여 답변의 정확성을 높입니다
  • 맥락 인식: 사용자의 대화 이력을 활용하여 자연스러운 다중 턴 대화를 지원합니다
  • 확장성: 새로운 지식 소스의 추가가 용이한 모듈화된 아키텍처입니다
  • 추적 가능성: 모든 응답에 대한 참고 출처를 제시하여 신뢰성을 보장합니다

1.2 핵심 기술 스택

계층 기술 설명
웹 프레임워크 FastAPI 고성능 Python 웹 프레임워크
LLM OpenAI GPT / Claude API 대규모 언어 모델
벡터 데이터베이스 Pinecone / Qdrant 의미 기반 정보 검색
임베딩 OpenAI Embeddings 텍스트 벡터화
관계형 DB PostgreSQL 구조화된 데이터 저장
캐싱 Redis 응답 속도 개선
메시지 큐 RabbitMQ / Celery 비동기 작업 처리
모니터링 Prometheus + Grafana 시스템 메트릭 수집 및 시각화

1.3 서비스 목표

기술적 목표:
  • 응답 시간: 평균 < 2초 (P95 < 5초)
  • 가용성: 99.9% uptime
  • 정확도: 사용자 만족도 > 85%

1.4 주요 사용 사례

  • 고객 지원: FAQ 및 제품 문서 기반의 자동 고객 응답
  • 내부 지식 검색: 직원들의 빠른 정보 접근
  • 연구 보조: 학술 자료 검색 및 요약
  • 콘텐츠 생성: 기사, 보고서 작성 지원

2.2 아키텍처 계층 설명

2.2.1 프리젠테이션 계층 (Presentation Layer)

클라이언트 요청을 수신하고 응답을 반환하는 계층입니다.

  • API Gateway: 모든 API 요청의 진입점 (Kong 또는 Nginx)
  • 인증/인가: JWT, OAuth2 기반 보안
  • 속도 제한: DDoS 방지 및 공정한 자원 배분
  • 요청/응답 변환: 프로토콜 변환 및 직렬화

2.2.2 비즈니스 로직 계층 (Business Logic Layer)

핵심 기능을 구현하는 계층입니다.

  • 대화 관리: 세션, 메시지 히스토리 관리
  • RAG 엔진: 정보 검색 및 생성
  • 데이터 검증: 입력 데이터 검증
  • 비즈니스 규칙: 도메인 로직

2.2.3 데이터 접근 계층 (Data Access Layer)

데이터 저장소와의 상호작용을 담당합니다.

  • ORM: SQLAlchemy를 이용한 DB 추상화
  • 벡터 DB 클라이언트: 의미 검색 쿼리
  • 캐시 클라이언트: Redis 연결
  • 파일 시스템: 문서 및 이미지 저장

2.3 마이크로서비스 구조

선택적으로 마이크로서비스 아키텍처로 확장할 수 있습니다:

  • 인증 서비스: 사용자 인증/인가
  • 대화 서비스: 메시지 및 세션 관리
  • 검색 서비스: 벡터 검색 및 유사도 계산
  • 생성 서비스: LLM 호출 및 응답 생성
  • 분석 서비스: 사용자 행동 분석

2.4 배포 아키텍처

클라우드 환경 (AWS, GCP, Azure 등)
  • 쿠버네티스 클러스터: 컨테이너 오케스트레이션
  • Auto Scaling: 부하에 따른 자동 확장
  • 로드 밸런싱: 트래픽 분산
  • 네트워크 격리: VPC, 보안 그룹
  • 저장소: S3/GCS (객체 저장소), RDS (관리형 DB)

특징:

3.2.3 RAG 엔진

검색과 생성을 통합하는 핵심 엔진입니다.

RAG 파이프라인:
  1. 사용자 쿼리 임베딩
  2. 벡터 DB에서 관련 문서 검색
  3. 검색된 문서로 프롬프트 구성
  4. LLM에 프롬프트 전달
  5. 생성된 응답 반환

3.2.4 캐싱 계층

Redis를 이용한 다층 캐싱 전략입니다.

  • 쿼리 결과 캐싱: 자주 검색되는 쿼리
  • 임베딩 캐싱: 이미 계산된 임베딩
  • 생성 결과 캐싱: 동일 질문에 대한 응답
  • 세션 캐싱: 사용자 세션 정보

3.2.5 비동기 작업 처리

Celery를 이용한 백그라운드 작업 처리입니다.

  • 문서 임베딩 계산
  • 대량 데이터 임포트
  • 주기적인 캐시 갱신
  • 분석 리포트 생성

4. RAG 파이프라인

4.1 RAG 개념 설명

Retrieval-Augmented Generation (RAG)는 대규모 언어 모델(LLM)이 학습 데이터에 없는 새로운 정보에 대한 질문에 답할 수 있도록 하는 기술입니다. 검색(Retrieval)과 생성(Generation)을 결합하여 더 정확하고 신뢰할 수 있는 응답을 생성합니다.

4.2 RAG 파이프라인 상세

4.2.1 문서 준비 단계 (Indexing)

학습 데이터를 벡터 DB에 저장하기 위한 준비 과정입니다.

단계별 프로세스:
  1. 문서 수집: PDF, 웹 페이지, DB 등에서 문서 수집
    • 웹 크롤링: Beautiful Soup, Selenium
    • PDF 파싱: PyPDF2, pdfplumber
    • DB 쿼리: SQLAlchemy ORM
  2. 전처리: 텍스트 정제 및 정규화
    • 특수 문자 제거
    • 공백 정규화
    • 언어 감지 및 필터링
  3. 청킹: 긴 문서를 작은 단위로 분할
    • 고정 크기 청킹: 문자 수 기반
    • 의미 기반 청킹: 문장/문단 단위
    • 오버래핑: 컨텍스트 연속성 보장
  4. 임베딩: 텍스트를 벡터로 변환
    • OpenAI Embeddings API (1536차원)
    • 또는 로컬 모델: sentence-transformers
  5. 저장: 벡터와 메타데이터를 DB에 저장
    • 벡터 DB: Pinecone, Qdrant, Weaviate
    • 메타데이터: PostgreSQL

4.2.2 쿼리 처리 단계 (Retrieval)

사용자 질문에 관련된 문서를 검색하는 과정입니다.

검색 절차:
  1. 쿼리 임베딩: 사용자 질문을 벡터로 변환
  2. 벡터 유사도 검색: 코사인 유사도로 가장 유사한 청크 선택
  3. 메타데이터 필터링: 시간, 카테고리 등으로 추가 필터링
  4. 재랭킹: BM25 또는 크로스 인코더로 결과 재정렬
  5. 결과 반환: 상위 K개 청크 반환 (보통 K=5-10)

4.2.3 응답 생성 단계 (Generation)

검색된 정보를 기반으로 최종 응답을 생성합니다.

생성 절차:
  1. 프롬프트 구성: 검색 결과를 기반으로 프롬프트 생성
  2. 컨텍스트 추가: 대화 히스토리 포함
  3. LLM 호출: OpenAI GPT 또는 Claude API 호출
  4. 응답 처리: 토큰 스트리밍, 형식화
  5. 출처 추적: 사용된 문서 정보 기록

4.3 고급 RAG 기법

4.3.1 Multi-Query RAG

여러 쿼리 변형으로 검색 범위를 확대합니다.

구현 방식:
  • LLM으로 원본 쿼리의 5-10개 변형 생성
  • 각 변형으로 벡터 검색 수행
  • 중복을 제거하고 통합

4.3.2 Hierarchical RAG

계층적 문서 구조를 활용합니다.

  • 문서 → 섹션 → 청크 단계적 검색
  • 먼저 문서 수준에서 검색, 필요시 더 상세한 청크 검색

4.3.3 Hybrid RAG

벡터 검색과 키워드 검색을 결합합니다.

  • 의미 기반 벡터 검색: 의도와 개념 캡처
  • 키워드 검색 (BM25): 정확한 용어 매칭
  • 결과 통합: RRF(Reciprocal Rank Fusion)

4.4 RAG 성능 최적화

4.4.1 청킹 전략

방식 장점 단점 추천
고정 크기 구현 단순 의미 무시 일반 텍스트
문장 단위 의미 보존 일관성 부족 대화식 콘텐츠
문단 단위 배경 정보 포함 청크 크기 불균등 기술 문서
의미 청킹 최적의 의미 보존 계산 비용 높음 구조화된 문서

4.4.2 임베딩 모델 선택

  • OpenAI Embeddings: 성능 우수, 비용 발생
  • Sentence Transformers: 무료, 로컬 실행, 약간의 성능 손실
  • Voyage AI: 긴 문맥 처리 최적화

4.4.3 검색 결과 개선

  • 재랭킹: 초기 검색 후 크로스 인코더로 재정렬
  • 필터링: 메타데이터 기반 사전 필터링
  • 앙상블: 여러 검색 방법 결합
  • 피드백 루프: 사용자 피드백으로 개선

5.2 세션 데이터 흐름

사용자 세션과 메시지 히스토리 관리 방식입니다.

세션 생명주기:
  1. 생성: 새 사용자 로그인 시 세션 생성
  2. 저장: Redis에 세션 데이터 캐싱, PostgreSQL에 영구 저장
  3. 메시지 추가: 각 턴마다 메시지 히스토리 업데이트
  4. 컨텍스트 유지: 최근 N개 메시지를 LLM 프롬프트에 포함
  5. 종료: 사용자 로그아웃 또는 타임아웃 시 세션 종료

5.3 메시지 흐름 (실시간)

WebSocket 기반 실시간 통신:
  1. 클라이언트가 메시지 전송
  2. 백엔드에서 즉시 메시지 ACK 반환
  3. 비동기로 RAG 처리 시작
  4. 생성된 응답을 스트리밍으로 클라이언트에 전송 (청크 단위)
  5. 최종 응답 완료 후 메타데이터 전송

5.4 데이터 저장소별 저장 정보

5.4.1 PostgreSQL

구조화된 관계형 데이터 저장:

  • users: 사용자 정보, 인증 정보
  • sessions: 대화 세션
  • messages: 메시지 히스토리
  • documents: 문서 메타데이터
  • usage_logs: API 사용량

5.4.2 Pinecone (벡터 DB)

의미 검색을 위한 벡터 저장:

  • 벡터: 문서 청크의 임베딩 (1536차원)
  • 메타데이터: 문서 ID, 소스, 생성일
  • 인덱스: 빠른 유사도 검색

5.4.3 Redis

캐싱 및 세션 저장:

  • 세션 데이터: 사용자 세션 (1시간 TTL)
  • 쿼리 결과: 자주 검색되는 쿼리 (24시간 TTL)
  • 임베딩 캐시: 계산된 임베딩 (7일 TTL)

5.5 에러 처리 및 재시도

에러 타입별 처리 전략:
  • 네트워크 에러: 지수 백오프로 3회 재시도
  • API 속도 제한: 요청 대기열 관리
  • 데이터베이스 에러: 트랜잭션 롤백 및 로깅
  • LLM 에러: 폴백 모델 사용

5.6 모니터링 및 로깅

모든 데이터 흐름 단계에서 로깅 및 모니터링이 수행됩니다.

메트릭 수집 방식 저장소 알림 조건
응답 시간 각 API 엔드포인트 측정 Prometheus P95 > 5초
에러율 Exception 로깅 ELK Stack 에러율 > 1%
토큰 사용량 LLM API 응답 PostgreSQL 예산 초과
캐시 히트율 Redis 명령 Prometheus 히트율 < 50%

6. API 명세

6.1 API 설계 원칙

  • RESTful: 표준 HTTP 메서드와 상태 코드 사용

6.3 응답 코드

코드 설명 예시
200 성공 GET 요청 성공
201 생성됨 리소스 생성 성공
202 수락됨 비동기 작업 시작
400 잘못된 요청 필수 필드 누락
401 인증 필요 토큰 누락 또는 만료
403 권한 없음 다른 사용자의 자원 접근
429 요청 초과 속도 제한 초과
500 서버 에러 예기치 않은 에러

7.2 배포 프로세스

7.2.1 CI/CD 파이프라인

GitHub Actions 기반 자동화:
  1. 코드 푸시: Git에 코드 커밋
  2. 테스트: pytest 실행 (커버리지 > 80%)
  3. 린팅: pylint, black, mypy로 코드 검사
  4. 빌드: Docker 이미지 생성
  5. 푸시: ECR에 이미지 푸시
  6. 배포: ArgoCD로 k8s 업데이트 (GitOps)
  7. 검증: 헬스 체크 및 스모크 테스트
  8. 모니터링: 알림 설정 및 메트릭 수집

7.2.2 배포 전략

  • Blue-Green: 일부 트래픽을 새 버전으로 라우팅
  • Canary: 5% → 25% → 50% → 100% 점진적 배포
  • 로롤백: 문제 발생 시 이전 버전으로 즉시 복귀

7.3 모니터링 및 로깅

7.3.1 메트릭 수집 (Prometheus)

메트릭 의미 임계값 알림
http_request_duration_seconds 요청 응답 시간 P95 > 5초 경고
http_requests_total 전체 요청 수 분당 < 100 비정상 요청
http_request_errors_total 에러 요청 수 > 1% 심각
vector_db_query_time_ms 벡터 검색 시간 > 500ms 경고
cache_hit_ratio 캐시 히트율 < 50% 경고
db_connection_pool_usage DB 연결 풀 사용율 > 80% 경고

7.3.2 로깅 (ELK Stack)

Elasticsearch + Logstash + Kibana 스택으로 통합 로깅

  • 애플리케이션 로그: 요청, 에러, 디버그
  • 시스템 로그: 쿠버네티스, 인프라 메트릭
  • 보안 로그: 인증, 권한 변경
  • 비즈니스 로그: 사용자 행동, API 호출

7.3.3 알림 규칙

심각도별 알림:
  • 치명: Slack #critical, PagerDuty 호출
  • 경고: Slack #alerts, 이메일
  • 정보: 로그 기록만

7.4 성능 최적화

7.4.1 데이터베이스 최적화

  • 인덱싱: 자주 검색되는 필드에 인덱스
  • 쿼리 최적화: EXPLAIN ANALYZE로 느린 쿼리 식별
  • 연결 풀링: PgBouncer로 연결 재사용
  • 레플리카: 읽기 전용 복제본으로 읽기 분산

7.4.2 캐싱 전략

  • L1 캐시: 애플리케이션 메모리 (로컬)
  • L2 캐시: Redis (분산 캐시)
  • L3 캐시: CDN (정적 콘텐츠)

7.4.3 리소스 최적화

  • CPU: 각 Pod 0.5 CPU, 버스트 최대 1 CPU
  • 메모리: 각 Pod 512MB, 최대 1GB
  • 스토리지: 임시 데이터는 ephemeral, 영구 데이터는 PVC

7.5 보안 관리

7.5.1 인증 및 권한

  • API 인증: JWT 토큰 기반
  • 데이터베이스 접근: IAM 역할 기반
  • RBAC: Kubernetes Role-Based Access Control

7.5.2 데이터 보안

  • 전송: TLS 1.3 암호화
  • 저장: at-rest 암호화 (KMS)
  • 민감 정보: Secrets 관리 (Sealed Secrets)

7.5.3 취약점 관리

  • 의존성 스캔: Dependabot로 자동 스캔
  • 이미지 스캔: Trivy로 Docker 이미지 검사
  • 정기 보안감사: 분기별 보안 감시

7.6 재해 복구 (DR)

7.6.1 백업 전략

  • 데이터베이스: 일일 전체 백업 + 시간당 증분 백업
  • 벡터 DB: 일일 스냅샷
  • 문서: S3 Cross-Region Replication
  • 보존 정책: 30일 보유

7.6.2 복구 계획

시나리오 RTO RPO 실행 방법
Pod 재시작 1분 0 자동 (Kubernetes)
노드 장애 5분 0 자동 (클러스터 재스케줄링)
DB 장애 15분 1시간 RDS 자동 페일오버
지역 장애 1시간 1시간 다른 지역 클러스터 활성화
